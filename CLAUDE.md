# CLAUDE.md

리조트 예약·판매 시스템 프로젝트 작업 지침서. Claude Code는 이 문서를 프로젝트 컨텍스트로 사용한다.

## 프로젝트 개요

리조트 내 타프 텐트·캐빈 하우스·트레일러를 고기세트 포함 상품으로 판매하는 **온라인 예약 시스템**이다. 실제 운영되는 서비스이며, 예상 트래픽은 월 1,000명 이하. 동시 접속 부하는 거의 없으므로 성능 최적화보다 **결제-예약 정합성**과 **슬롯 중복 예약 방지**가 최우선 과제다. 고기세트가 포함된 상품 특성상 노쇼 방지를 위해 전액 선결제를 받는다.

## 기술 스택

- **프론트엔드/백엔드:** Next.js (App Router) 풀스택, TypeScript
- **데이터베이스/인증:** Supabase (PostgreSQL + Supabase Auth)
- **결제:** 토스페이먼츠 직접 연동 (서버 측 결제 승인 검증 필수)
- **알림:** 문자(카카오 알림톡 기반 예정). 발송 대행 서비스는 미정 — 추상화 레이어로만 구현
- **호스팅:** Vercel
- **언어:** 모든 코드는 TypeScript. `any` 사용 지양

## 도메인 모델

예약 단위는 **시설 × 날짜 × 부(시간대)** 조합이다.

### 시설 (총 13동)
- 타프 텐트 8동
- 캐빈 하우스 4동
- 트레일러 1동

### 운영일
- 금요일, 토요일만 운영

### 시간대 (2부제)
- 1부: 17:00 ~ 19:00
- 2부: 19:30 ~ 21:30

→ 하루 최대 13동 × 2부 = 26슬롯. 주당(금·토) 최대 52슬롯. 각 슬롯은 독립적으로 예약 가능하며, **동일 슬롯에 대한 중복 예약은 DB 트랜잭션/제약으로 원천 차단**해야 한다.

## 결제 정책

- **전액 선결제:** 예약 시 100% 결제 완료되어야 예약 확정
- 결제 승인은 반드시 **서버 측에서 토스페이먼츠 API로 재검증** (클라이언트 응답만 신뢰하지 않음)
- 결제 성공 ↔ 예약 레코드 생성은 하나의 트랜잭션으로 묶어 정합성 보장
- 취소/환불은 토스페이먼츠 결제 취소 API 연동 (관리자 페이지에서 처리)

## 관리자 페이지 (`/admin`)

운영자가 코드 수정 없이 모든 운영을 제어할 수 있어야 한다. Supabase Auth로 접근 보호.

- 예약 현황 조회 (날짜·시설·부별 캘린더/리스트 뷰)
- 슬롯 열기/닫기 (특정 날짜 휴무 처리, 임시 마감 등)
- 시설·상품·가격 관리
- 예약 수동 취소 및 환불 처리 (토스 결제 취소 연동)
- 매출·정산 조회
- 알림 발송 이력 조회

## 알림 (문자)

- 발송 대행 서비스(솔라피/NHN Cloud/알리고 등)는 **미정**
- 발송 로직을 **인터페이스(추상화 레이어)로 분리**하여, 추후 어떤 서비스를 붙여도 구현체만 교체하면 되도록 설계
- 발송 시나리오: 예약 확정 / 방문 전 리마인더 / 취소 안내

## 디렉터리 구조 (제안 — 구현하며 확정)

```
/app              # Next.js App Router
  /(public)       # 고객용 예약 페이지
  /admin          # 관리자 페이지 (인증 보호)
  /api            # 결제 검증, 웹훅 등 서버 라우트
/lib              # 공용 로직 (supabase 클라이언트, 결제, 알림 인터페이스)
  /payments       # 토스페이먼츠 연동
  /notifications  # 알림 추상화 레이어 (구현체 교체 가능)
/types            # 공용 타입 정의
/supabase         # 마이그레이션, RLS 정책
```

## 코딩 규칙

- TypeScript strict 모드. 타입 우선
- 환경 변수(토스 시크릿 키, Supabase 서비스 키 등)는 절대 클라이언트에 노출하지 않음. 서버 전용으로 관리
- 금액·결제 관련 로직은 반드시 서버에서 검증
- Supabase RLS(Row Level Security) 정책으로 데이터 접근 제어
- 커밋 전 타입 체크/린트 통과

## 하지 말아야 할 것

- 클라이언트 결제 응답만 믿고 예약을 확정하지 말 것 (반드시 서버 재검증)
- 슬롯 가용성 확인과 예약 생성을 분리된 비원자적 작업으로 처리하지 말 것 (race condition 발생)
- 시크릿 키를 `NEXT_PUBLIC_` 접두사로 노출하지 말 것
- 알림 발송 서비스 SDK를 비즈니스 로직에 직접 하드코딩하지 말 것 (추상화 레이어 경유)

## 환불 / 취소 규정 (확정)

결제 금액 기준, 이용일 자정(KST) 기준으로 계산. 상세는 `docs/booking-flow.md`.

- 이용 2일 전까지: 100% 환불 (차감 0%)
- 이용 1일 전: 50% 환불
- 이용 당일: 환불 없음
- 노쇼: 당일 취소와 동일(환불 없음)

## 미정 / 추후 결정

- 문자 발송 대행 서비스 선택 (솔라피/NHN Cloud/알리고 등)
- 카카오 알림톡 템플릿 사전 승인 절차
- 도메인, 사업자 정보
- pending_payment 타임아웃 확정값 (기본 10분 제안)
- 슬롯 자동 생성 주기 (몇 주치 미리 open 할지)
- 간편결제(카카오페이·네이버페이) 추가 여부 (필요 시 포트원 도입 재검토)
